# Memex vs MemPalace — 设计理念对比

## 核心定位

| | MemPalace | Memex Phase 2 |
|---|---|---|
| **目标** | maximize recall（找回正确答案） | maximize utility（找回最有用的答案） |
| **设计理念** | 越简单越好 | 越智能越好 |
| **适合场景** | benchmark 刷分 | 真实用户场景 |

## Benchmark 结果

| 配置 | R@5 | R@10 | NDCG@10 |
|------|------|------|---------|
| MemPalace (turn-level) | 96.6% | 98.2% | — |
| Memex (turn-level, 500题) | 91.8% | 94.0% | 80% |
| Memex (session-level, 500题) | 85.8% | 92.2% | 78.9% |

差距原因：粒度不同（MemPalace ~550条/题，Memex ~53条/题）导致召回率差异。

## 机制对比

| 机制 | MemPalace | Memex Phase 2 |
|------|-----------|----------------|
| **语义蒸馏** | ❌ 无（所有内容都存） | ✅ 过滤闲聊/情绪/确认，只存有价值内容 |
| **混合检索** | ❌ 纯向量相似度排序 | ✅ 0.55×相似度 + 0.20×重要性 + 0.15×新鲜度 + 0.10×频率 |
| **记忆修订** | ❌ 不支持（append-only） | ✅ 稳定性状态机，矛盾时降级 |
| **衰减遗忘** | ❌ 无 | ✅ decay ≥ 0.7 归档，CONSTRAINT/USER_MODEL 永不归档 |
| **矛盾检测** | ❌ 无 | ✅ 否定模式 + 行为冲突检测，自动降低 importance |
| **记忆合并** | ❌ 无 | ✅ 相似度 > 0.85 自动合并 |
| **记忆类型** | 纯 chunks | 6 种分类（CONSTRAINT/USER_MODEL/STRATEGY/SYSTEM_PATTERN/BELIEF/JOURNAL） |
| **接口** | API | CLI + Agent Skill |

## 为什么 MemPalace 不用这些机制？

MemPalace 的成功靠三点：
1. **原文存储** — 不丢失信息
2. **turn-level 粒度** — 细节丰富
3. **好 embedding** — BGE 多语言模型

这三个已经能让 R@5 达到 96.6%。加更多机制反而可能引入噪声、降低分数。

## 为什么 Memex 要做这些？

LongMemEval 评价的是 **recall**（能否找回正确答案），不是 **utility**（找到的答案有没有用）。

真实场景中：

> 你问"用户之前对什么不满？" 
> - 一条三个月前的闲聊（向量相似度一般，但直接回答了问题）
> - 一条上周的技术讨论（向量相似度高，但没直接回答）

混合检索能把"三个月前的闲聊"排到前面，因为它虽然老，但**直接有用**。

## Phase 2 机制评测困境

这些 Phase 2 机制（蒸馏、混合检索、修订、衰减、合并）没有公开 benchmark：

| 机制 | 评测难点 |
|------|----------|
| 蒸馏 | 取决于"什么有价值"，没有 ground truth |
| 混合检索 | 评价用户价值，不是 R@5 准确性 |
| 修订/矛盾 | 需要有冲突的成对记忆场景 |
| 衰减 | 需要时间维度的数据 |
| 合并 | 需要有重复/相似内容的场景 |

**可能的评测方向**：
1. **人工评估**：构造场景 + 人工判断排序质量
2. **代理指标**：用蒸馏后的存储量 reduction ratio、矛盾检测触发次数等
3. **用户反馈**：集成反馈机制，收集真实使用数据

这些机制的价值最终需要在**真实使用**中验证，而不是在 benchmark 上刷分。
