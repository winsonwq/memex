# MemPalace 记忆宫殿 — 深度调研报告

> 调研时间：2026-04-09
> 项目地址：https://github.com/milla-jovovich/mempalace

---

## 一、记忆宫殿是如何打造的？

### 核心理念

不依赖 LLM 做"什么值得记忆"的判断，直接存储所有原文，用结构（宫殿）来组织，用语义搜索来召回。

### 存储架构

```
ChromaDB (向量数据库) ─── 存储 verbatim 原文（drawers 抽屉）
SQLite ─── 存储时序知识图谱（entity + relationship + validity window）
```

### 宫殿层级结构

```
Wing（翅膀）  = 人 / 项目  ← 顶层隔离
  └─ Room（房间） = 具体话题（auth-migration、ci-pipeline）
       └─ Closet（衣柜） = AAAK 压缩后的摘要（指向原文）
            └─ Drawer（抽屉） = 原始 verbatim 内容
```

**Hall（走廊）** 是跨房间的语义连接（同类记忆类型）：
- `hall_facts` 决策
- `hall_events` 事件
- `hall_discoveries` 发现
- `hall_preferences` 偏好

**Tunnel（隧道）** 是跨 wing 的连接——同一个 room 名出现在不同 wing 时自动建立。

例如 `auth-migration` 在 `wing_kai`（个人）和 `wing_driftwood`（团队）中都出现，tunnel 就把两者连通了。

### 挖掘（Mine）方式

- `convos` 模式：对话导出，按 exchange pair 分块，检测人/项目/偏好/情绪信号
- `project` 模式：代码 + 文档，检测实体和项目
- 提取用 Regex 模式（不是 LLM）：偏好检测 16 种表达模式

### 四层记忆栈（按需加载）

| Layer | 内容 | Token 数 | 何时加载 |
|-------|------|---------|---------|
| L0 | Identity（身份定义） | ~50 | 始终 |
| L1 | Critical facts（团队/项目/偏好） | ~120 | 始终 |
| L2 | Room recall（按 wing/room 过滤的召回） | ~200-500 | 按需 |
| L3 | Deep search（全量语义搜索） | 无上限 | 按需查询 |

## 二、优势、性能、recall 能力 & 代价

### Benchmark 核心数据（LongMemEval R@5，500 题）

| 模式 | 得分 | API 调用 | 成本 |
|------|------|---------|------|
| **Raw ChromaDB（纯 verbatim）** | **96.6%** | **0** | **$0** |
| Hybrid v4 + Haiku rerank | 100% | ~500 次 | ~$0.50 |
| Mem0 | ~85%（生产）/ ~45%（ConvoMem） | 需要 | $19-249/月 |
| Supermemory | ~85%（生产）/ ~99%（实验） | 需要 | — |
| Mastra | 94.87% | GPT-5-mini | API 费用 |
| Zep | ~85% | 需要 + Neo4j | $25/月+ |

### 核心 Insight

Raw 模式是最高分，但不需要任何 API、任何 LLM、任何云服务。

大多数记忆系统都在"用 LLM 决定记什么"这件事上内耗，而 verbatim + 结构化搜索本身就足够强。

**为什么 verbatim 赢了**：当 LLM 提取"user prefers PostgreSQL"并丢弃原文时，它丢失了 *why*、*alternatives considered*、*tradeoffs discussed*。MemPalace 全部保留，搜索时原文就在那里。

### Palace 结构的贡献（+34% R@10 的来源）

```
搜所有 closets:           60.9%  R@10
限定 wing 搜索:           73.1%  (+12%)
wing + hall 过滤:         84.8%  (+24%)
wing + room 过滤:         94.8%  (+34%)
```

宫殿结构不是装饰——是元数据过滤带来检索提升，和向量相似度分数相乘。

### 代价 & 局限

1. **AAAK 是有损的**：README 最初声称"30x lossless"，社区打脸后发现 AAAK 在 LongMemEval 上只有 84.2%（vs raw 96.6%）
2. **词汇不匹配问题**：问"yoga classes"，但原文说"I went this morning"——纯语义搜索会漏
3. **单人/单会话 assistant 问题**（R@5 ~93%）：问"assistant 上次说了什么"比问"user 偏好"更难召回
4. **No native multi-agent sharing**：每个 AI 实例维护自己的 palace，团队共享需要额外的同步机制
5. **100% 分数有教学考题之嫌**：最后三个改进是在看完成卷子后针对错题做的——团队自己也坦白了

## 三、从 Harness Engineering 角度看：记忆能做到什么程度？

### MemPalace 对个人 Agent 的价值

| 维度 | 当前 Agent 状态 | MemPalace 能补什么 |
|------|--------------|-----------------|
| **上下文知道** | 靠 CLAUDE.md 手工注入 | ✅ L2/L3 自动按 wing/room 加载 |
| **知道自己是谁** | 靠 SOUL.md / IDENTITY.md | ✅ L0 identity.txt + L1 auto-generated |
| **知道规则是什么** | 靠 CLAUDE.md | ❌ 没有覆盖 |
| **知道当前状态** | 靠对话历史（有限窗口） | ✅ 跨 session 记忆 |
| **知道团队成员** | 不知道 | ✅ wing 结构 |

### MemPalace 的根本局限

它是被动的存储层，agent 仍然需要主动调用 `mempalace_search` 工具。不像 kairo 那样把记忆和推理引擎更紧密地耦合在一起——没有 inference-time 的记忆推理，只有点击式的检索。

### 对 Team 的价值与局限

**价值**：wing 结构天然适配团队场景，可以追踪跨项目、跨人的决策历史和偏好。

**局限**：团队共享目前是断裂的——没有原生的团队级记忆广播或冲突解决机制。

### 记忆能做到的程度总结

**可以做到**：
- 跨 6 个月、跨多个项目的 semantic search
- 多层级的记忆召回（个人 wing vs 团队 wing）
- 时序推理（"Kai 什么时候从 Orion 离开的？"）

**做不到**：
- inference-time 主动推断记忆——必须先问对问题
- 团队级别的记忆一致性

## 四、评价与启示

**最值得关注的一点**：MemPalace 证明了"不靠 LLM 做记忆提取"这个方向是可行的，而且简单方法比复杂方法更强。96.6% 的 raw baseline 本身就是最大的 insight——整个记忆系统行业都在过度工程化。

**对阿秋工作流的参考价值**：如果在使用 Claude Code，可以考虑把 MemPalace 作为长期记忆的存储层，配合 kairo 的 harness 框架——前者负责"记住"，后者负责"推理和行动"。

**团队场景下**：需要有人维护 palace 的结构（wing/room 分类），一旦分类乱了，tunnel 的价值就大幅下降。
