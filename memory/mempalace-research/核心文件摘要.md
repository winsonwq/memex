> 来源：https://github.com/milla-jovovich/mempalace（2026-04-09 clone）

## palace_graph.py — 核心图结构

### 数据模型

```python
# ChromaDB metadata 结构
{
    "room": "auth-migration",      # 房间名（具体想法）
    "wing": "wing_kai",            # 归属 wing
    "hall": "hall_facts",          # 记忆类型走廊
    "date": "2026-01-15",          # 日期
    "source_file": "...",          # 原始文件
}

# 节点: {room: {wings: set, halls: set, count: int}}
# 边: [{room, wing_a, wing_b, hall, count}]
```

### 关键函数

- `build_graph()` — 从 ChromaDB metadata 构建节点和边
- `traverse(start_room, max_hops)` — BFS 遍历，从一个房间出发走 N 步
- `find_tunnels(wing_a, wing_b)` — 找连接两个 wing 的 room（tunnel）
- `graph_stats()` — 图统计：总 room 数、tunnel 数、rooms per wing

### Tunnel 的形成逻辑

当同一个 room 名出现在 ≥2 个不同的 wing 时，自动形成 tunnel：
```
room "auth-migration" in wing_kai AND wing_driftwood → tunnel
```

## knowledge_graph.py — 时序知识图谱

### 数据模型

```python
# SQLite 表结构
entities: id, name, type, properties, created_at
triples: id, subject, predicate, object, valid_from, valid_to, confidence, source_closet, source_file

# 示例 triple
add_triple("Kai", "works_on", "Orion", valid_from="2025-06-01")
add_triple("Maya", "assigned_to", "auth-migration", valid_from="2026-01-15")
```

### 关键能力

- **时序有效性**：每个 fact 有 `valid_from` / `valid_to`，支持"某时间点什么是真的"查询
- **invalidate()**：事实过期时自动标记，不再返回（但不删除历史）
- **query_entity(as_of=date)**：时间旅行查询

### vs Zep Graphiti

| Feature | MemPalace | Zep |
|---------|-----------|-----|
| Storage | SQLite | Neo4j |
| Cost | Free | $25/mo+ |
| Temporal validity | Yes | Yes |
| Self-hosted | Always | Enterprise only |
| Privacy | Everything local | SOC 2, HIPAA |

## layers.py — 四层记忆栈

### Layer 0 — Identity

```python
# 读取 ~/.mempalace/identity.txt
# 内容示例：
I am Atlas, a personal AI assistant for Alice.
Traits: warm, direct, remembers everything.
People: Alice (creator), Bob (Alice's partner).

```

### Layer 1 — Essential Story

- 从 ChromaDB 拉取最多 15 个最重要/最新的 drawers
- 按 room 分组，生成 ~500-800 token 的紧凑摘要
- 每次 wake-up 时自动生成

### Layer 2 — On-Demand

```python
# 按 wing/room 过滤检索
retrieve(wing="myapp", room="auth-migration", n_results=10)
```

### Layer 3 — Deep Search

```python
# 全量语义搜索
search("pricing change", n_results=5)
```

## dialect.py — AAAK 有损压缩

### 格式示例

```
wing|room|date|source_file
0:ALC+BOB|auth_migration|"decided to switch to Clerk"|3.5|determ|DECISION
```

### 组件

- **Entity codes**：Alice → ALC，Bob → BOB（减少重复）
- **Emotion codes**：vul, joy, fear, love, hope 等
- **Flags**：ORIGIN, CORE, SENSITIVE, PIVOT, DECISION, TECHNICAL
- **Key quote**：从原文提取的最重要的一句话

### 重要警告

AAAK 是**有损摘要**，不是可逆压缩。原始内容不可从 AAAK 重建。
benchmark 96.6% 的分数来自 **raw mode**（不经过 AAAK），AAAK 模式只有 84.2%。

## entity_detector.py — 自动实体检测

### 两阶段检测

**Pass 1**: 扫描文件，提取出现 3+ 次的 capitalized 名词候选

**Pass 2**: 对每个候选打分，分类为 person / project / uncertain

### Person 信号（权重高）

- 对话标记：`"{name} said"`, `"{name} asked"`
- 人称动词：`{name} decided`, `{name} felt`, `{name} wanted`
- 代词临近：`she/her` 在 name 附近 3 行内
- 直接称呼：`hey {name}`, `thanks {name}`, `hi {name}`

### Project 信号（权重高）

- 项目动词：`building {name}`, `ship {name}`, `deploy {name}`
- 版本号：`MemPalace v2`, `mempalace-core`
- 代码引用：`mempalace.py`, `mempalace.yaml`

### 分类阈值

- 需要 **至少两种不同信号类别** 才能 confident 分为 person
- person_ratio ≥ 0.7 + 两种信号类型 + ps ≥ 5 → person
- 否则 uncertain

## benchmarks/BENCHMARKS.md — 关键发现

### LongMemEval 核心结论

1. **Raw verbatim + ChromaDB 默认 embeddings = 96.6% R@5**（无需 LLM，无需 API key）
2. 所有"改进"都是在试图弥补 vocabulary gap 的问题
3. LoCoMo 上 raw 60.3%，但加了 hybrid scoring + Sonnet rerank 后达到 100%
4. Temporal-inference（跨多 session 的时序推理）是从 46% → 100% 提升最大的类别

### Score 进展

| Mode | R@5 | NDCG@10 | LLM | Cost |
|------|-----|---------|-----|------|
| Raw ChromaDB | 96.6% | 0.889 | None | $0 |
| Hybrid v1 | 97.8% | — | None | $0 |
| Hybrid v2 | 98.4% | — | None | $0 |
| Hybrid v2 + Haiku | 98.8% | — | Haiku | ~$0.001 |
| Hybrid v3 + Haiku | 99.4% | 0.983 | Haiku | ~$0.001 |
| **Hybrid v4 + Haiku** | **100%** | 0.976 | Haiku | ~$0.001 |
| **Hybrid v4 held-out 450q** | **98.4%** | 0.939 | None | $0 |

### 诚实承认的问题

1. **100% 是教学考题**：最后 0.6% 的提升（3 个 question-specific fixes）是在看完卷子后针对错题做的
2. **LoCoMo 100% 用 top-k=50**：超过了每个 conversation 的 session 数量（19-32），检索步骤实质被绕过
3. **30x lossless compression 夸大**：AAAK 是有损的，不是真正的压缩
